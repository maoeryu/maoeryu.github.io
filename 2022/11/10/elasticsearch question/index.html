使用elasticsearch做分页查询时,当查询记录超过10000时报错

因为es数据是存储在不同分片上的,假设取的页数较大时(深分页),像是请求第 20 页,es 不得不取出所有分片上的第 1 页到第 20 页的所有文档,并做排序,最终再取出结果.
假设你有 16 个分片,则需要处理 16 * (20 + 10) 条记录后做一次全局排序,而当索引非常非常大(千万或亿)时,会非常消耗 CPU 和内存资源,所以为了保护机器,es使用 index.max_result_window:10000 这个设定作为保护措施.

解决办法:修改最大限制

但这种方法只是暂时解决问题,当数据量越来越大,分页也越来越深,还是会出问题的,业务上尽量避免这种情况,一般也没有需要翻页这么深的需求.

[script] Too many dynamic script compilations within, max: [1/1m]

错误原因

elastic每分钟内编译script的个数有限制,如果在脚本中将变量硬编码时,elastic每次都会编译.
而将变量作为参数传入时,则不会重复编译.

错误重现

a.将集群script个数限制改为1,手动更改变量来触发错误

b.创建索引,插入数据

c.不断更改age值,重现错误

解决方法

a.动态修改script.max_compilations_rate值,提高限制.
b.将变量作为参数传入(建议使用).

测试后,无错误出现.

No Compatible Fields: The xxx index pattern does not contain any of the following field types: date

错误原因

在绘制kibana日期类图表时,因其只支持date类型的字段作为x轴,当时间戳字段类型写入为long时,图表无法展示,提示错误.

解决方法

利用Scripted fields,在不改变elastic索引中原始数据的前提下,增加一个字段作为时间戳.

测试

在kibana页面,依次点击management->index patterns,选择索引,再选择Scripted fields->Add scripted field,添加新字段updatedtime,type选date,format选Date,pattern为YYYY-MM-DD HH:mm:ss,Script为doc['ts'].value
